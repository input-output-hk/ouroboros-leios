# T27 threat: mempool partitioning via network control

> Mempool partitioning via network control → inconsistent mempools, conflicting EBs. Mitigation is
> limited because of the directional flow difference (txs flow upstream, block data downstream).

A realistic adversarial test of any system requires a setup as close to the real environment as
possible. The more synthetic, regular, and simplified the environment, the less the tested
functionality tells you about the real one — in the worst case, simulations on a too-simple
substrate are not just meaningless but actively misleading.

Therefore, a node topology resembling the Cardano mainnet has been built. See
[topology-v4](../data/simulation/pseudo-mainnet/topology-v4.md) for further details.

## Edge simulation

Current version of `sim-rs` simulates message-and-byte dynamics over a static graph. The edge set
itself doesn't evolve. That means peer discovery and hot/warm/cold tiers are not implemented. As the
consequence, simulations (attacks) in T27 scenario with `sim-rs` can be done only with static edges.
A some degree of the dynamic of the connection will be achievable once `sim-rs` will have the
ability to manage the edges at runtime. This is work in progress. This can bring the test
environment closer to the real one, but still network recovery from the attack might be different.
Such recovery in real system is likely more optimistic than the situation observed in test
environment, as real peer sharing protocol has ability to self-heal the network.

However, even with simulated edges' dynamic, this is still only an approximation of the real system
and the limitations of th used tools need to be taken into account as well.

## T27 description

T27 targets **upstream tx propagation**: an adversary with network control prevents tx broadcasts
from crossing a partition boundary, so different block producers see different mempool subsets and
emit EBs that reference different (and sometimes mutually conflicting) tx sets. Voting quorum on
either side fails or wastes votes on EBs the other side will never adopt.

Note on what does _not_ count as T27: starting two sub-networks from genesis with no shared chain
history is **not** T27 — it's two parallel Cardano deployments. T27 requires a healthy converged
network that is _then_ partitioned, so the interesting effects (conflicting EBs sharing a common
chain prefix, quorum failure across a previously-working voting set, reconvergence after heal) are
observable.

## How T27 can be evaluated

T27 effects show up at scale. The faithful test is **sim-rs against `topology-v4-mainnet.yaml`**
(2,685 nodes, real stake/region distribution). sim-rs has no live UI — output is JSONL events,
analysed post-hoc with the notebooks in `analysis/`.

## Scenarios

Each scenario is an A/B against the same `topology-v4-mainnet.yaml` baseline (intact network, no
attack), comparing certification rate, EB conflict count, and per-region mempool divergence over
time.

### S1. TX-direction partition via `late_tx_attack` _(sim-rs; partial today)_

`sim-rs/sim-core/src/config.rs` `late_tx_attack` already supports a selection + `tx_start_time` /
`tx_stop_time` window. What works today is the **explicit-node-list** selector:

```yaml
late_tx_attack:
  attackers: { nodes: ["node-0", "node-7", "node-12", ...] }
  attack_probability: 1.0
  tx_start_time: 300.0 # warmup phase: 5 min healthy
  tx_stop_time: 900.0 # 10 min attack window, then heal
```

`NodeSelection` (config.rs) currently has only `Nodes(HashSet<String>)` and `StakeFraction(f64)` —
there is **no `Region` variant**. Sim-rs also doesn't carry country / region / ASN on
`NodeConfiguration`: the topology YAML's `RawNode` has `location: [lon, lat]` (or a cluster name)
and nothing else. The richer geographic metadata (`cc`, `region`, `provider`, `asn`, `pool_id`)
lives in the `*.yaml.meta.json` sidecar produced by the topology-viewer pipeline; sim-rs does not
read it.

To select attackers by region the patch is:

1. Extend `RawNode` with optional `cc` / `region` fields (or teach the loader to read the sidecar);
2. Carry them into `NodeConfiguration`;
3. Add a `Region(String)` variant to `NodeSelection` and the matching arm in `Topology::select`.

Until then, hand-list the attacker node IDs from the meta.json (or pre- compute them with `jq` / a
notebook).

Result: post-warmup, the attacker nodes' txs stop reaching the rest of the network. Block producers
on either side of the partition include only the txs that did reach them; EBs that reference the
same parent RB but disjoint tx sets cannot both certify.

Limitation: covers only the **tx-direction** of the partition.

### S2. Network-layer partition _(sim-rs only; not started, ~600 LOC)_

The causally-honest reproduction: a new `partition-scenarios` config block adds time-windowed edge
drops in `NetworkCoordinator::schedule_message` so the partition lives in the network layer, not in
node behaviour. Predicate-based selection (e.g. `from-continent != to-continent`) and explicit
(`from = node-X, to = node-Y`) supported. Time-window fields would mirror S1 (`start_time` /
`stop_time`). Telemetry events `PartitionStarted` / `PartitionHealed` written to the JSONL stream so
post-hoc analysis can correlate them with the certification dip.

Scope and phasing pending a design doc. Until that lands, S1 alone covers the tx-direction of the
partition with the caveat noted in §Edge simulation (behavioural attack, not link cut).

### S3. Heal dynamics _(S1 or S2 with a finite window)_

Setting `tx_stop_time` (S1) / `stop_time` (S2 once implemented) makes the attack window finite.

**What "heal" actually means here.** For S1 the topology is static throughout — what changes at
`tx_stop_time` is the per-node behaviour switch: attackers stop withholding txs and resume honest
forwarding over the same connections that were always there. For S2 (once implemented) the heal
would be a literal edge re-enable in `NetworkCoordinator`. Either way there is no peer discovery,
peer re-selection, or governor-driven recovery — those mechanisms aren't simulated (see
§Edge simulation).

Observable post-heal: chain re-convergence (Praos longest-chain resolves any forks the attack
created), mempool re-flooding (honest tx diffusion catches up via the still-existing edges),
conflicting EBs on the abandoned fork falling out of the certifiable window and getting orphaned.
This is the piece T27 in particular cares about — Linear Leios is designed to behave gracefully
here, and the simulation should show the recovery curve back to baseline. Bear in mind the caveat
above: real-world recovery via peer-sharing / governor would likely be faster than what the
simulation shows.

## Observing the attack

sim-rs writes a JSONL event stream — no live UI. Analyse with a notebook under
`analysis/sims/<run>/`; `analysis.ipynb` in any of the existing runs is the right template. The
signals that matter for T27:

- **Certification rate over time, banded by region.** Pre-attack: both regions certify at the
  steady-state baseline (~37% under CIP-0164 defaults). Inside the attack window: certification dips
  sharply (each side reaches only ~half the committee, falling below the 75% quorum threshold).
  Post-heal: recovery curve back to baseline — that recovery shape is the Linear Leios "graceful
  degradation" property worth measuring.
- **EB-conflict count.** Count slots where ≥ 2 EBs reference the same parent RB but disjoint tx
  sets. Should be ≈ 0 pre-attack, spike inside the window, decay after heal.
- **Per-region mempool divergence.** At each `EBGenerated` event, log the EB's tx set; aggregate by
  issuer-region; compute Jaccard distance between EU-issued and US-issued EB tx sets over a sliding
  window. Pre-attack ≈ 0, attack window ≈ 1, post-heal back to ≈ 0.
- **`MessageDropped` / `PartitionStarted` / `PartitionHealed` events** (would only be emitted by S2,
  which is not yet implemented) — correlate the certification dip with the partition window in the
  chart's time axis.
